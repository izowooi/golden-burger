# Golden Papaya 증액과 꼬리손실 위험

> 작성 기준일: 2026-07-16
>
> 목적: Final Five 전략의 증액 판단 근거를 보존한다. 이 문서는 수익성 입증이나
> `$100` 주문 승인이 아니다.

## 결론부터

- 현재 A/B는 주문 `$5`, 동시 포지션 20개, event당 1개를 유지한다.
- 두 잡의 진입 문턱은 모두 **Gamma 유동성 `$10,000` 이상 + 최근 24시간 거래량
  `$2,000` 이상**으로 올린다. 72시간 잡과 24시간 잡의 유일한 차이는 시간 상한이다.
- Papaya 자체 연구 archive는 유동성 `$1,000`/거래량 `$0`을 계속 수집한다. 운영 진입
  문턱에서 탈락한 시장도 첫 0.95 관측과 반사실 분석에 남겨야 하기 때문이다.
- 이 메타데이터 문턱만으로 `$100` 주문이 안전해지지는 않는다. `$100` 전환 전에는 실제
  CLOB 주문장 깊이, 예상 평균 체결가, spread·slippage, 부분체결 처리를 검사하는 별도
  실행 로직이 필요하다.
- `max_positions=20`은 누적 거래 횟수가 아니라 **한 Papaya job DB에서 동시에 열려 있는
  주문·포지션의 상한**이다. 100개나 200개로 임의 확대하지 않는다.

## 1. 작은 이익과 큰 꼬리손실의 비대칭

YES를 가격 `p`에 사서 1.00으로 해결되면 원금 대비 비용 전 수익률은
`(1 - p) / p`다. 반대로 YES가 0으로 해결되면 해당 원금은 전부 손실이다.

| 진입가 | 정상 해결 시 총수익률 | 전액 손실 1건을 복구하는 동일 금액 승리 수 |
|---:|---:|---:|
| 0.95 | 5.26% | 19.0건 |
| 0.97 | 3.09% | 32.3건 |
| 0.98 | 2.04% | 49.0건 |

따라서 “0.98에 사서 확정적으로 2%를 먹는다”는 표현은 정확하지 않다. 실제 성공확률이
시장가격과 같은 98%라면 비용 전 기대값부터 0이고, spread·fee·미체결·redeem 지연까지
넣으면 음수가 된다. 수익이 있으려면 실제 성공확률이 실행가격과 모든 비용을 합친 손익분기
확률보다 높아야 한다.

0.90 stop도 손실을 보장하는 가격이 아니라 **매도 시도 신호**다. 0.90에서 전량 체결된다는
가장 낙관적인 경우에도 0.95 진입은 약 -5.26%, 0.97 진입은 약 -7.22%다. 얇은 bid,
급격한 gap, 부분체결 또는 resolution 직행이 발생하면 실제 손실은 더 커질 수 있다.

## 2. 이미 0.95 이상인 시장과 0.97 초과 gap을 제외하는 이유

현재 전략이 검증하려는 사건은 단순히 “고확률 시장을 산다”가 아니라 다음과 같은
**관측 가능한 첫 상향 교차**다.

```text
직전 저장값 < 0.95 <= 현재 저장값 <= 0.97
직전과 현재의 간격 <= 30분
그보다 앞선 0.95 이상 관측 없음
```

- 첫 실행에서 이미 0.96이면 실제로 언제, 어떤 가격으로 0.95를 넘었는지 증명할 수 없다.
  이를 포함하면 cold start 시점에 따라 표본이 달라진다.
- 0.94에서 다음 관측이 0.98이면 0.95~0.97에서 실행 가능한 시간이 있었는지 증명할 수
  없다. 0.98을 0.97에 체결된 것으로 소급하면 look-ahead와 가상 체결이 된다.
- 이 제외 규칙은 해당 시장이 수익성이 없다는 뜻이 아니다. “최초 발견 고확률 시장” 또는
  “0.97 초과 gap 진입”은 별개의 가설과 cohort로 사전 등록해 검증할 수 있다. 현재의
  72시간 대 24시간 A/B에 동시에 섞으면 어느 변경이 결과를 만들었는지 알 수 없다.

## 3. 왜 현재 실행 방식 그대로 증액하면 안 되는가

현재 진입 필터의 `liquidity`와 `volume24hr`는 Gamma 시장 메타데이터다. 실제 주문 순간
`0.97 이하 ask에 몇 주가 있는지`를 뜻하지 않는다. Polymarket의 표시 가격도 실행가격을
보장하지 않으며 큰 주문은 여러 가격 수준을 소진해 평균 체결가를 움직일 수 있다.

Papaya는 주문 직전에 fresh best ask를 확인하고 그 가격에 GTC 주문을 낸다. 이 방식에는
다음 증액 공백이 있다.

1. best ask **한 가격 수준의 수량**이 주문 금액보다 작은지 사전에 막지 않는다.
2. 전체 ask depth로 계산한 예상 VWAP과 최대 slippage 제한이 없다.
3. GTC가 일부만 체결되고 나머지가 오래 남는 경우의 규모별 취소 정책이 없다.
4. `max_positions`는 Papaya job DB 기준이고, 같은 지갑의 다른 job·전략·수동 포지션까지
   합산한 wallet-wide 원금 상한이 아니다.
5. 0.90 stop은 trigger일 뿐이므로 증액할수록 exit depth 부족의 절대 손실액이 커진다.

공식 주문장 문서도 midpoint와 실제 실행가격을 구분하고, 큰 주문은 depth를 확인해야 한다고
설명한다. 참고: [Prices & Orderbook](https://docs.polymarket.com/concepts/prices-orderbook),
[Orderbook](https://docs.polymarket.com/trading/orderbook),
[Get order book](https://docs.polymarket.com/api-reference/market-data/get-order-book).

## 4. 유동성·거래량 기본값 결정

### 선택값

```text
POLYBOT_MIN_LIQUIDITY=10000
POLYBOT_MIN_VOLUME_24H=2000
```

이 값은 임의로 새로 만든 숫자가 아니라 기존 월간 반사실 grid의 사전 등록점이다. 미래
`$100` 주문과 비교하면 주문액은 표시 유동성의 최대 1%, 최근 24시간 거래량의 최대 5%다.
이 비율은 `$1,000/$0`보다 명백히 보수적이며, `$5,000/$500`에서 `$100`이 각각 2%와
20%가 되는 것보다 낫다.

다만 이것은 **메타데이터 1차 gate**일 뿐이다. 표시 유동성의 정의, token별 bid/ask 위치,
진입 순간의 변화를 반영하지 않으므로 “무리 없이 체결”을 보장하지 않는다.

2026-07-16 11:36 UTC의 단일 공개 Gamma snapshot을 sanity check한 결과, 당시 strict
Yes/No·YES 0.95~0.97·잔여 0~72시간 시장 12개 중 각 문턱을 만족한 수는 다음과 같았다.

| 유동성 / 최근 24h 거래량 | 해당 시장 수 |
|---|---:|
| `$1k / $0` | 12 |
| `$5k / $500` | 3 |
| `$10k / $2k` | 1 |
| `$20k / $5k` | 0 |

이 표는 한 시점의 현재 가격대일 뿐 first-crossing 발생률이나 장기 성과의 백테스트가 아니다.
`$10k/$2k`가 표본을 크게 줄일 수 있다는 운영상 대가를 보여준다. 그래서 낮은 문턱의
archive를 유지하고, 최소 4주와 terminal event-cluster 30개 전에는 문턱을 다시 튜닝하거나
수익성을 주장하지 않는다.

## 5. `$100` 전환 전에 필요한 실행 gate

`$100`은 아래 항목이 구현되고 simulation 및 소액 live에서 검증된 뒤에만 별도 cohort로
올린다.

1. 주문 직전 전체 CLOB book을 가져온다.
2. `0.97` 이하 ask의 누적 executable size와 예상 VWAP을 계산한다.
3. 목표 수량보다 충분한 depth가 없으면 주문하지 않는다. 초기 연구값으로 `1.25 × 목표
   수량`을 검토할 수 있지만, 이 배수도 실측 후 사전 등록해야 한다.
4. 최대 허용 VWAP·spread·slippage를 설정하고 초과하면 fail closed한다.
5. FOK/FAK 또는 짧은 만료 후 잔량 취소처럼 부분체결 정책을 명시하고, 재시도로 중복 주문이
   생기지 않게 한다.
6. 계정 전체 available balance/allowance, open order, 다른 job을 포함한 총 원금 상한을
   주문 전에 확인한다.
7. actual 성과는 exact order ID의 confirmed fill size/price/fee로만 확정한다.

권장 단계는 다음과 같다.

| 주문액 | 판단 |
|---:|---|
| `$5` | 현재 A/B·증거 수집 단계 |
| `$10` | 최소 evidence gate 통과 후 별도 cohort 후보 |
| `$100` | 위 depth/VWAP/총원금 gate 구현 후에만 후보 |
| `$1,000` | 현재 실행 구조로 승인 불가; 별도 실행·위험 설계 필요 |
| `$5,000` | 현재 실행 구조로 승인 불가; 시장충격·분할실행까지 별도 검증 필요 |

## 6. 동시 20포지션 제한의 정확한 의미

코드는 해당 `--job`의 SQLite에서 다음 상태의 `Trade` 행을 센다.

```text
PENDING_BUY + HOLDING + PENDING_SELL
```

즉 20은 다음이 아니다.

- 지금까지 누적한 거래 횟수
- 주식 수나 token 수
- 모든 전략을 합친 지갑 전체 포지션 수
- 서로 다른 Jenkins job을 합친 전역 상한

한 job에서 동시에 잠길 수 있는 진입 원금의 단순 상한은
`buy_amount × max_positions`다.

| 1건 주문액 | 20포지션 원금 | A/B 두 job 합계 단순 원금 |
|---:|---:|---:|
| `$5` | `$100` | `$200` |
| `$10` | `$200` | `$400` |
| `$100` | `$2,000` | `$4,000` |
| `$1,000` | `$20,000` | `$40,000` |
| `$5,000` | `$100,000` | `$200,000` |

20개 제한이 필요한 이유는 전략의 승률이 높아 보여도 손실이 독립적이지 않기 때문이다.
같은 경기·리그·뉴스·가격 충격의 여러 event가 함께 실패할 수 있고, `event당 1개` 제한은
같은 event 내부만 막을 뿐 category 간 상관을 막지 못한다. 또한 열린 주문이 많을수록
자본 잠김, stop 유동성 부족, API 대사와 unknown submission의 운영 위험이 커진다.

100개/200개가 안전하다는 근거는 현재 없다. 포지션 상한은 시장 후보 수가 아니라 사전에
정한 위험 예산에서 역산해야 한다.

```text
job별 최대 포지션 = floor(job별 허용 진입 원금 / 1건 주문액)
```

실제 운영값은 여기에 wallet-wide 원금 상한, category/event 상관 한도, terminal loss
예산을 함께 적용한 값 중 가장 작은 값이어야 한다. 현재는 `$5 × 20 = job당 $100`의 작은
falsification 예산을 유지한다. 두 job이 같은 지갑을 사용한다면 로컬 20개 제한이 합산되지
않으므로 운영자는 최대 40개의 open state가 생길 수 있음을 별도로 통제해야 한다.

## 7. 현재 A/B 고정안

| 항목 | 72시간 job | 24시간 job |
|---|---:|---:|
| 주문액 | `$5` | `$5` |
| 진입 확률 | `0.95~0.97` | `0.95~0.97` |
| stop | `0.90` | `0.90` |
| 유동성 / 최근 24h 거래량 | `$10k / $2k` | `$10k / $2k` |
| snapshot 최대 간격 | `30분` | `30분` |
| 동시 포지션 / event | `20 / 1` | `20 / 1` |
| 잔여시간 | `0h < t <= 72h` | `0h < t <= 24h` |

두 job은 다른 `--job`, DB, 가능하면 다른 계정을 쓰고 같은 시점·cadence로 실행한다.
비교 기간에는 확률 상한, 거래량, 포지션 수 등 다른 축을 함께 바꾸지 않는다.

## 8. 증액 중단 조건

다음 중 하나라도 해당하면 금액을 올리지 않는다.

- `polybot-retro audit --strict`의 `CRITICAL` 또는 `HIGH`
- terminal event-cluster 30개 또는 최소 4주 미달
- confirmed fill/fee/redeem 증거 누락
- 비용 후 event-cluster 기대값이 0 이하
- stop trigger 대비 confirmed SELL fill률·slippage가 허용 범위를 벗어남
- 특정 event/category를 제외하면 성과가 사라짐
- full-book depth 검사가 목표 주문액을 반복적으로 수용하지 못함

이 전략에서 증액은 “승률이 높아 보인다”가 아니라 **꼬리손실, 실행 가능성, 상관 집중을
포함한 실제 비용 후 손익이 증명되었다**는 별도의 운영 결정이어야 한다.
