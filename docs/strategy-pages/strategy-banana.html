golden-banana — Momentum Golden Cross

확률 85~97% 구간의 고확률 outcome을, 자체 축적한 5분 스냅샷 기반 momentum이 상승 중일 때만 매수하는 golden cross 전략.

유형 모멘텀 편승상태 운영 중핵심 편향 underreaction (정보 확산 지연)

핵심 수식

momentum = (최신 확률 - 가장 오래된 확률) / 스냅샷 수    # 5분 주기 자체 스냅샷, 끝점 기울기
short = momentum(최근 3개 ≈ 15분),  long = momentum(최근 72개 ≈ 6시간)

BUY : 0.85 ≤ p ≤ 0.97 (상한 포함)
      AND ( short - long ≥ +0.02  AND  long > 0        # golden cross
            OR 스냅샷 < 6개 AND short > 0 )              # cold-start fallback
SELL: p ≥ 0.97 │ P&L ≤ -10% │ P&L ≥ +7% │ short - long ≤ -0.02 (dead cross)

전략 개요

Golden Banana는 Polymarket에서 확률 85~97% 구간의 고확률 outcome을 "모멘텀이 상승 중일 때만" 매수하는 momentum 전략 봇이다. 구버전(golden-apple 계열)의 "80% 매수 → 90% 매도" 정적 임계값 전략을 개선한 것으로, 5분 주기의 Jenkins 실행마다 전체 시장 확률을 SQLite market_snapshots 테이블에 자체 축적하고, 이 스냅샷으로 단기(15분=3 스냅샷)·장기(6시간=72 스냅샷) momentum을 계산해 golden cross(단기-장기 ≥ +2%p/스냅샷) 발생 시 진입한다.

청산은 4중 조건(확률 97% 도달, 진입가 대비 +7% 이익실현, -10% 손절, dead cross)으로 다변화했다. 주문은 CLOB v2에 midpoint 가격의 GTC limit order로 실행되며, 한 번 거래한 시장은 영구히 재거래하지 않는다.

매수·매도 조건

가격 이력 수집 — 핵심 메커니즘: 외부 price-history API를 쓰지 않고 봇이 스스로 이력을 만든다. 매 사이클 Phase 0에서 Gamma 전량 조회(100개씩 페이지네이션, offset ≥ 5,000 안전 한도) 후 min_liquidity 이상 + 비스포츠 시장마다 높은 쪽 확률을 market_snapshots에 1행 INSERT(실측 로그 기준 사이클당 약 30~35행). 판단 시 최신 82개(long_window+10)를 조회하며, 스냅샷 간격 = Jenkins cron 간격(5분 가정)이지만 timestamp 간격은 검증하지 않고 개수 기반 윈도우만 사용한다. 7일 지난 스냅샷은 Phase 4에서 삭제.

momentum 계산: (최신 확률 - 가장 오래된 확률) / len(snapshots) — 이동평균(MA) cross가 아니라 양 끝점 기울기(rate of change) 비교다. 단기는 최근 3개(3개 미만이면 진입 불가), 장기는 최근 72개(72개 미만이어도 6개 이상이면 있는 만큼 사용, 6개 미만이면 None). 장기가 None이면 cold-start fallback으로 단기 momentum > 0 만으로 진입을 허용하며(short_momentum_positive), 실운영 로그(20260205)에서 실제 매수는 전부 이 경로였다.

이용하는 심리

  • 정보 확산 지연 / underreaction: 사실상 결정된 사건도 대중 전체가 인지·베팅하기까지 시간이 걸려 85~97% 구간에서 100%로 서서히 수렴한다는 가정. 15분 momentum이 6시간 추세를 앞지르는 순간 = "새 정보가 지금 가격에 반영되는 중"으로 보고 그 물결에 편승한다.
  • favorite-longshot bias의 역이용: 대중은 낮은 확률(longshot)에 과대 베팅하고 고확률(favorite)을 상대적으로 저평가하는 경향이 있어, 고확률 outcome을 사는 쪽이 기대값상 유리하다는 전제.
  • 군집 행동(herding) 편승: golden cross 진입은 상승 cascade 초입에 올라타고, dead cross 청산은 반대 방향 herding이 시작되면 빠지겠다는 설계. 정적 임계값 전략이 당하던 "고점 매수 후 뉴스 반전" 손실을 momentum 반전 감지로 방어한다.

강점

  • 전략과 인프라의 분리: 진입/청산 판단이 momentum.py 한 곳에 있어 sibling 봇에서 이 파일만 교체하면 된다 (golden-cherry가 실제로 이 방식으로 파생됨).
  • 자체 스냅샷 축적: 외부 이력 API 의존 없이 5분 해상도 확률 시계열을 확보. entry_reason/exit_reason/momentum 컬럼으로 사후 전략 분석(SQL)이 가능.
  • 다중 청산 조건: 구버전의 "매도 조건 하나, 손절 없음" 문제를 threshold/take_profit/stop_loss/dead_cross 4중으로 해결.
  • 운영 방어 로직: 429 Retry-After 존중, 422 페이지네이션 종료 처리, tick 반올림 + clamp, midpoint 재검증(rapid_jump), 최소 5주 검증, job별 DB/로그 격리, simulation mode 전 구간 지원.
  • 가시성: 매 사이클 momentum 분석 요약 표(진입/제외 사유, 단기/장기/차이, 스냅샷 수)를 로그로 남겨 파라미터 튜닝 근거가 된다.

약점 (허점)

  • 의도한 시그널(golden cross)이 사실상 발화 불가능: 임계값 0.02는 "스냅샷당 +2%p" 기울기 차이 — 단기 윈도우 3개 기준 15분에 약 6%p 이상 급등해야 한다. 85~97% 구간에서 극히 드물며, 실로그의 diff는 ±0.007 이하. 실제 매수는 전부 cold-start fallback 경로였다.
  • cold-start fallback이 과도하게 관대: 스냅샷 6개 미만이면 15분간 +0.33%p(1 tick의 1/3)만 움직여도 매수. DB 초기화 직후·유동성 경계 신규 통과·7일 정리 이후에는 사실상 "85~97% + 방금 안 떨어짐" 전략으로 퇴화.
  • 개수 기반 윈도우, 시간 미검증: Jenkins 중단·지연 시 "15분" 윈도우가 수 시간을 커버해도 그대로 계산.
  • 체결 가정 오류: GTC limit 주문을 접수 즉시 HOLDING/COMPLETED로 기록. (a) 매수 미체결이면 유령 포지션, (b) 손절 매도 미체결이면 DB상 손절 완료지만 실제로는 계속 보유 — 급락장에서 stop loss가 보장되지 않는다.
  • 해결된 시장의 포지션 영구 잔류: 시장 해결 후 orderbook 소멸 → midpoint 404 → execute_sell 영구 실패. 온체인 redeem이 되어도 DB realized_pnl 미반영으로 통계 왜곡.
  • take_profit 도달 불가 구간: 0.94 이상 진입 시 +7% 목표가 1.0 초과 → 수학적으로 불가능. 0.97 정확히 진입 시 다음 사이클 threshold 매도로 즉시 청산되는 churn 엣지도 존재.
  • 스포츠 키워드 필터 과잉 차단: "game", "match", "win the", "beat", "score" 같은 일반 단어로 비스포츠 시장(수상·선거 관련 질문)까지 배제. 기회비용이 크다.
  • momentum 계산의 통계적 취약성: 양 끝점 2개 값만 사용해 끝점 노이즈에 민감. 분모가 구간 수(n-1)가 아닌 개수(n)라 기울기를 (n-1)/n만큼 과소평가. 장기 분모가 유동적이라 단기와 단위가 어긋난다.
  • 영구 1회 거래 제한: condition_id당 평생 1회. rapid_jump로 한 번 skip되면 영구 차단 — 수개월짜리 시장의 정당한 재진입 기회 상실.
  • 자금 관리 부재: max_positions=-1 + 고정 5 USDC. 잔고 확인·일일 매수 한도·변동성 기반 sizing 없음.
  • Gamma 전량 조회 2회/사이클: Phase 0과 Phase 2가 같은 조회를 중복 호출 (API 비용 2배).
  • scripts/simulate.py DB 경로 버그: sim 거래가 trades_sim.db가 아닌 해당 job의 trades.db에 기록된다 (polybot run --simulate는 정상).
  • 미사용/불일치 코드: DataAPIClient·SlackNotifier는 dead code, data_api_client의 User-Agent가 GoldenApple-PolyBot/1.0으로 잔존, excluded_categories는 env override 없음.
  • 시뮬레이션의 낙관 편향: midpoint 100% 체결 가정 — spread·슬리피지·수수료 미모델링으로 sim P&L이 실거래보다 항상 좋게 나온다.

개선 반영처

  • golden-grape — 도달 가능한 드리프트 신호: 사실상 발화 불가능한 golden cross 임계값(0.02/스냅샷 = 15분에 +6%p 필요)과 과도하게 관대한 cold-start fallback을, 실제로 도달 가능한 drift 신호로 재설계해 반영.

A/B 검증 계획

① 시뮬레이션: polybot run --simulate는 실거래와 동일 경로에서 주문만 mock(trades_sim.db 분리). 단, midpoint 100% 체결 가정이라 sim P&L은 낙관 편향(약점 14)이 있고, scripts/simulate.py는 DB 경로 버그(약점 12)가 있으므로 --simulate 플래그 경로를 사용한다. ② 소액 실거래: yaml buy_amount_usdc: 5.0으로 건당 노출 제한. ③ 판단 후 증액: 매 사이클 로그의 momentum 분석 요약 표(진입/제외 사유, 단기/장기 diff, 스냅샷 수)와 entry_reason/exit_reason 컬럼 집계로 "golden cross 경로 대비 cold-start 경로"의 실적을 비교해 임계값 재보정 여부를 결정한다.
