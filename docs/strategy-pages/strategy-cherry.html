Golden Burger strategy contract

🍒 golden-cherry Resolution Momentum

75~92%의 Yes 후보를 5일 이내 또는 주문 가능한 인플레이에서 포착하고, 손절·익절·트레일링으로 관리하는 현재 운영 계약.

이름을 혼동하지 않는다 주문 자금과 conditional token을 보유한 계좌는 golden-banana이고, 실행 코드·SQLite DB·run audit의 전략 이름은 golden-cherry다.

현재 전략 계약

공통 진입:
  Gamma tradeable = active=true AND closed=false
                    AND enableOrderBook=true AND acceptingOrders=true
  0.75 ≤ 현재 CLOB midpoint ≤ 0.92
  실효 유동성 ≥ max(POLYBOT_MIN_LIQUIDITY,
                     POLYBOT_BUY_AMOUNT / POLYBOT_MAX_ORDER_LIQUIDITY_RATIO)

시간 진입:
  비스포츠      0h < endDate - now ≤ 120h
  스포츠 경기 전 0h < gameStartTime - now ≤ 120h
  스포츠 경기 중 POLYBOT_ALLOW_IN_PLAY=true AND Gamma tradeable

청산 우선순위:
  1. P&L ≤ -8%                    stop_loss
  2. P&L ≥ +10%                   take_profit
  3. 현재가 < 최고가 × (1 - 5%)   trailing_stop
  4. endDate 잔여시간 ≤ EXIT_HOURS time_exit

현재 POLYBOT_EXIT_HOURS=0 → 4번 시간 청산은 비활성

전략의 가설은 고확률 favorite이 결과 확정에 가까워질수록 1.0 쪽으로 수렴한다는 것이다. 현재 Jenkins 운영은 POLYBOT_YES_ONLY=true와 run --yes-only를 함께 사용하므로 outcome index 0, 일반적인 binary 시장의 Yes만 평가한다. POLYBOT_SELL_THRESHOLD라는 이름은 오해하기 쉽지만 실제로는 신규 BUY 상한이며 매도 신호가 아니다.

92% 초과를 사지 않는 이유는 수익 보장이 아니라 손익 비대칭 제어다 98¢ 진입의 만기 최대 총수익은 비용 전 약 2.04%인 반면 패배하면 원금 대부분을 잃을 수 있다. 한 번의 전액 손실을 복구하려면 비용 전에도 약 49회의 98→100¢ 성공이 필요하다. 현재 코드는 주문 직전 가격이 92%를 넘으면 rapid_jump로 같은 job DB에서 영구 제외한다. 이 정책의 기회비용은 회고에서 별도로 측정해야 하며, 경험적으로 최적이라고 단정하지 않는다.

시간 조건을 정확히 읽는 법

비스포츠

endDate까지 0시간 초과 120시간 이하인 신규 후보만 통과한다.

스포츠 경기 전

endDate가 아니라 gameStartTime까지 0시간 초과 120시간 이하인지 본다.

스포츠 경기 중

ALLOW_IN_PLAY=true이고 Gamma가 계속 주문 가능하다고 보고하면 시간창과 별도로 진입할 수 있다.

  • ENTRY_HOURS_MIN=0은 시작 뒤까지 0시간을 포함한다는 뜻이 아니다. 경기 전/비스포츠 경계는 실제로 0 < 잔여시간이다.
  • 경기 시작 5분 전 차단 규칙은 현재 계약에 없다. 과거의 POLYBOT_GAME_START_BUFFER_MINUTES는 제거됐고 지금 export해도 읽지 않는다.
  • gameStartTime은 kickoff 시각일 뿐, 경기 종료까지 남은 분이나 “마지막 5분”을 알려주지 않는다. 종료 임박만 골라 사려면 별도 live score/clock 데이터가 필요하다.
  • REJECT_SPORTS_WITHOUT_GAME_START=true이면 스포츠인데 시작 시각이 없을 때 endDate로 추측하지 않고 제외한다.
시간창은 신규 BUY 필터다 0~120시간 밖에 있는 기존 보유 포지션을 그 이유만으로 팔지 않는다. 기존 포지션에는 손절·익절·트레일링이 계속 적용되고, 현재 EXIT_HOURS=0이라 시간 청산은 발생하지 않는다.

신규 진입 파이프라인

#검사현재 동작
1시장 전수 스캔Gamma keyset cursor를 끝까지 순회한다. 완주하지 못한 partial sweep을 정상 유니버스로 간주하지 않고 run audit에 attestation을 남긴다.
2주문 가능 상태active=true, closed=false, enableOrderBook=true, acceptingOrders=true를 모두 요구한다.
3유동성정적 $50,000과 주문액/0.2%의 동적 하한 중 더 큰 값을 적용한다.
4side·확률YES-only에서는 index 0 가격이 0.75~0.92인지 본다. 양끝은 포함한다.
5시간·스포츠 단계비스포츠/endDate, 경기 전/gameStartTime, 인플레이 허용 규칙을 구분한다.
6주문 직전 재검증CLOB midpoint와 스포츠 단계를 다시 본다. 92% 초과는 rapid_jump 영구 skip, 75% 미만은 이번 cycle만 skip한다.
7노출 한도건당 하드캡, 포지션 수, open 요청 원금, cycle 신규 건수, 주문/유동성 비율을 재검증한다.
8주문·증거GTC 주문을 execution ledger에 기록하고 주문·trade·fill을 후속 cycle에서 대사한다. 불확실한 token/side는 격리한다.

시장당 1회 제한은 전 세계 지갑 기준이 아니라 현재 job의 SQLite DB 기준이다. 같은 지갑을 다른 job/DB에서 함께 쓰면 중복 진입 가능성을 별도로 통제해야 한다.

주문 규모와 유동성

실효 최소 유동성
  = max(정적 최소 유동성, 주문금액 / 최대 주문·유동성 비율)
  = max($50,000, 주문금액 / 0.002)

$100 주문    → 최소 $50,000
$1,000 주문  → 최소 $500,000
$10,000 주문 → 최소 $5,000,000
  • 현재 $100 주문은 MAX_BUY_AMOUNT_USDC=100 하드캡과 일치한다.
  • MAX_OPEN_NOTIONAL_USDC=5000이므로 $100씩이면 이론상 약 50개에서 원금 한도가 먼저 작동한다. MAX_POSITIONS=100과는 별도다.
  • MAX_NEW_POSITIONS_PER_CYCLE=5는 5분마다 최대 5건이라는 burst 제한이지 총 포지션 제한이 아니다.
  • 주문액을 늘릴 때는 건당 하드캡과 open 원금 한도를 의도적으로 함께 검토해야 한다.
Gamma liquidity는 호가창 depth가 아니다 $10,000 주문을 위해 숫자만 $5,000,000으로 올려도 충분하지 않다. 실제 ask/bid depth, 예상 slippage, 부분체결, 이벤트 상관 노출을 측정한 뒤 단계적으로 증액해야 한다.

Jenkins 실행 예시

스케줄과 shell은 별도 입력란이다 Jenkins Build periodically의 Schedule에는 H/5 * * * *만 넣고, 아래 내용은 Execute shell에 넣는다. 동시 빌드는 비활성화한다. 
#!/bin/bash
set +x

# Jenkins Credentials Binding이 아래 두 변수를 이미 주입해야 한다.
# POLYMARKET_PRIVATE_KEY
# POLYMARKET_FUNDER_ADDRESS
# POLYMARKET_SIGNATURE_TYPE은 기존에 주문이 정상인 값을 유지한다.

export POLYBOT_LIFECYCLE_MODE=active
export POLYBOT_YES_ONLY=true

export POLYBOT_ENTRY_HOURS_MIN=0
export POLYBOT_ENTRY_HOURS_MAX=120
export POLYBOT_EXIT_HOURS=0

export POLYBOT_GAME_START_FILTER_ENABLED=true
export POLYBOT_ALLOW_IN_PLAY=true
export POLYBOT_REJECT_SPORTS_WITHOUT_GAME_START=true

export POLYBOT_BUY_AMOUNT=100
export POLYBOT_MAX_BUY_AMOUNT_USDC=100
export POLYBOT_MIN_LIQUIDITY=50000
export POLYBOT_MAX_ORDER_LIQUIDITY_RATIO=0.002
export POLYBOT_MAX_POSITIONS=100
export POLYBOT_MAX_OPEN_NOTIONAL_USDC=5000
export POLYBOT_MAX_NEW_POSITIONS_PER_CYCLE=5

cd ./golden-cherry

/Users/jongwoopark/.local/bin/uv sync --frozen
/Users/jongwoopark/.local/bin/uv run python ./main.py config
/Users/jongwoopark/.local/bin/uv run python ./main.py run --yes-only

main.py config를 호출하는 이유

이 명령은 거래하지 않는 선택적 preflight다. 환경변수·YAML·코드 기본값을 합쳐 필수 인증값, 확률 범위, 시간창, 하드캡과 노출 한도의 모순을 검사하고 secret을 제외한 resolved config와 DB 경로를 출력한다. Gamma/CLOB API나 주문은 호출하지 않지만 data/default/가 없으면 디렉터리를 만들 수 있다.

run도 같은 설정 검증을 다시 하므로 필수는 아니다. 설정 변경 직후 Jenkins 로그에서 실제 적용값을 확인하려면 유지하는 편이 좋다. config와 run에 --job을 쓴다면 반드시 같은 값을 써야 기존 DB와 포지션을 이어간다.

주문 증거와 남은 한계

  • Gamma keyset 완주와 qualifying membership을 run audit에 기록한다.
  • CLOB submission, order status, trade association, partial fill을 execution ledger에 기록·대사한다.
  • 불확실한 이전 주문은 같은 token/side 신규 주문을 격리하며, 증거 정합성을 지킬 수 없으면 fail closed한다.
  • 기존 SQLite에는 additive migration으로 스포츠 시간·단계 evidence 컬럼을 보충한다.
  • 매도 시 실제 conditional-token 잔고에 맞춰 수량을 줄이고 최소 단위 dust를 처리한다.
GTC 접수는 체결이 아니다 orderID, live, accepted만으로 실제 성과를 확정하지 않는다. 회고의 실현 손익은 order_fills.status='CONFIRMED'인 실제 size·price와 fee coverage로 계산하고, legacy 또는 미대사 구간을 분리한다.
  • midpoint 한계: 스캔·청산 판단의 midpoint는 실제 BUY ask/SELL bid와 다르며 spread·slippage를 반영하지 않는다.
  • 해결과 redeem: endDate 경과만으로 해결을 단정하지 않는다. resolution evidence와 redeem evidence는 분리해야 하며 현재 전략은 자동 redeem 회계를 완성하지 않았다.
  • 인플레이 시간: Gamma 메타데이터만으로 실제 경기 종료까지 남은 시간을 알 수 없다.
  • 상관 노출: 100개 포지션을 허용해도 같은 경기·선거의 파생 시장은 독립 표본이 아니다. 총 원금 한도만으로 이벤트 집중 위험이 사라지지 않는다.
  • 수익성 미확정: 75~92% favorite 수렴은 전략 가설이다. 증액은 confirmed-fill cohort, 실제 depth와 수수료를 포함한 회고 뒤에만 판단한다.

정밀 회고 절차는 golden-cherry 회고 가이드, 공통 성과 판정은 Evidence Contract, 실행 옵션은 Golden Cherry README를 따른다.
