👑 Golden Queen

Crown Momentum

YES가 0.90을 처음 상향 돌파하는 순간만 잡고, 0.98 목표와 0.85 절대 stop으로 관리한다. 스포츠를 기본 포함하며, 주문 접수와 실제 체결을 엄격히 분리한 Cherry 계승 전략이다.

0.90 → 0.94 진입 0.98 목표 · 0.85 stop 스포츠 기본 포함 기본 $5 · simulation

한눈에 보는 계약

진입 = strict standard binary YES + 저장된 직전 YES < 0.90 ≤ 현재 YES ≤ 0.94 + 0분 < 두 snapshot 간격 ≤ 15분 + 이전 60일에 YES ≥ 0.90 관측 없음 + 일반/경기 전: 0시간 < 남은 시간 ≤ 24시간 + 스포츠 경기 중: kickoff 후 ≤ 360분 + fresh spread ≤ 0.02 + 실제 ask depth ≥ 주문 수량 × 1.20 청산 = 미해결 YES ≥ 0.98 AND fresh bid ≥ 0.98 OR 미해결 YES ≤ 0.85 그 외 = resolution · redeemable/실제 redeem · CLOB fill을 별도 사실로 취급
스포츠는 환경변수를 설정하지 않아도 포함됩니다. gameStartTime이 있으면 경기 시작 전과 시작 후 360분까지 감시하고, 값이 없으면 endDate를 사용합니다. category 제외는 Gamma tag의 exact match이므로 제외가 필요할 때만 실제 tag를 POLYBOT_EXCLUDED_CATEGORIES에 comma로 명시합니다. 환경변수를 생략하면 스포츠를 포함합니다.

왜 이렇게 단순화했나

첫 교차 하나

고확률 시장을 계속 사지 않는다. 직전 저장값이 0.90 미만이고 현재가 처음 0.90을 넘은 한 번만 후보가 된다.

두 개의 출구

trailing stop과 time exit을 제거했다. 실행 가능한 0.98 목표 또는 0.85 절대 stop만 사용한다.

실제 호가 확인

Gamma 유동성 숫자만 믿지 않는다. 같은 CLOB snapshot에서 ask·bid·spread와 주문 수량 대비 depth를 검사한다.

체결 증거 우선

accepted/live는 체결이 아니다. exact full confirmed fill 전에는 BUY/SELL 모두 pending이다.

Cherry 데이터가 말해 준 것

Golden Queen은 “과거 수익이 입증된 최적 전략”이 아닙니다. Cherry의 2026-06-23~07-23 UTC strict audit가 CRITICAL 4, HIGH 6, MEDIUM 1로 실패했고 시장 snapshot은 0행이었습니다. 따라서 threshold를 과거 수익에 맞춰 최적화하지 않고, 실패 원인을 차단한 반증 가능한 신규 가설로 사전 등록했습니다.

gameStartTime은 경기 종료시각이 아닙니다. “종료 5분 전 매수”로 해석하지 않으며, 경기 중 가격 교차도 거래 대상입니다. sports 하나가 nba나 soccer 같은 별도 tag를 자동 포함하지 않습니다.

주문부터 실제 손익까지

Archive — cursor-complete sweep과 시장·event·snapshot을 원자적으로 저장하고 first-crossing lineage를 확정한다.
Preflight — fresh midpoint, best ask, spread, same-snapshot ask depth를 검증한다.
PENDING_BUY — GTC BUY 접수는 보유가 아니다. exact order의 full confirmed fill까지 기다린다.
HOLDING — confirmed BUY size·VWAP을 실제 포지션으로 기록하고 immutable 0.98/0.85를 적용한다.
PENDING_SELL → COMPLETED — SELL도 full confirmed fill, BUY/SELL size 일치, 양쪽 fee가 확인돼야 actual net P&L을 확정한다.

resolution 결과와 cash realization은 다릅니다. 현재 Queen은 resolution 결과를 기록하지만 redeemable 상태와 실제 redeem transaction은 아직 수집하지 않으므로, 이를 resolution이나 synthetic SELL에서 추정하지 않습니다.

원자적 sweep/archive 또는 RunAudit 실패는 cycle 전체를 중단합니다. 개별 시장의 event/lineage/book/depth 누락은 그 시장만 제외하고, 과거 주문의 개별 대사 오류는 해당 token_id × side 신규 주문만 격리합니다. archive_only도 시작 시 execution ledger 대사는 수행하지만 이후 주문과 trade-position mutation은 하지 않습니다.

최소 Jenkins shell

private key와 funder는 Jenkins Credentials Binding으로 주입합니다. shell에 실값을 적지 않고 secret을 참조하기 전부터 set +x를 사용합니다.

#!/bin/bash
set -euo pipefail
set +x

# POLYMARKET_PRIVATE_KEY / POLYMARKET_FUNDER_ADDRESS:
# Jenkins Credentials Binding
export POLYMARKET_SIGNATURE_TYPE=3
export POLYBOT_BUY_AMOUNT=5
export LOG_LEVEL=INFO

cd ./golden-queen
/Users/jongwoopark/.local/bin/uv sync --frozen
/Users/jongwoopark/.local/bin/uv run polybot config --job queen-live-24h
/Users/jongwoopark/.local/bin/uv run polybot run --live --job queen-live-24h

저장소 기본값은 simulation입니다. 실거래는 --live를 명시해야 합니다. 권장 timer는 H/5 * * * *이고 concurrent build는 비활성화합니다.

한 달 뒤 판정

  • STOP: strict audit 실패, 비용 후 event-cluster EV≤0, 12h/24h 모두 음수, 한 event를 빼면 edge 소멸.
  • KEEP: 최소 30일 + terminal event-effective n 30, full fill·fee evidence, paired/independent 결과 방향 일치.
  • SCALE: KEEP 이후에만 $5→$100 한 단계. A/B와 증액을 동시에 하지 않음.
